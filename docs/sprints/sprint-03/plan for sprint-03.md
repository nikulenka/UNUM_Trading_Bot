# 1. Цель спринта

Цель этого спринта состоит в том, чтобы реализовать первый самостоятельный компонент конвейера машинного обучения для торговой системы, а именно Docker-упакованную задачу подготовки исторических данных, которую можно запускать независимо от основного приложения.

Этот компонент будет подготавливать исходный рыночный набор данных, который затем будет использоваться для:
- инженерии признаков
- генерации меток
- обучения модели XGBoost
- экспериментов по проверке сигналов

Цель спринта, это создание основы данных.

Прежде чем обучать какую-либо модель машинного обучения, системе нужен надежный и воспроизводимый исторический набор данных.

Пайплайн обучения модели будет зависеть от:
- согласованной истории свечей OHLCV
- стабильных временных диапазонов
- идемпотентных повторных запусков
- проверенного охвата данных
- процесса, изолированного от живого торгового приложения

Без этого последующая работа по ML будет строиться на нестабильных предположениях об исходных рыночных данных.

Этот спринт и создает такую основу.

# 2. Планируемый результат

К концу спринта в проекте появится самостоятельная задача подготовки ML-данных, которая:
- запускается в Docker как отдельный сервис
- по умолчанию отключена
- может быть запущена вручную при необходимости
- загружает исторические рыночные данные свечей за настраиваемый диапазон
- сохраняет данные в PostgreSQL
- поддерживает безопасные повторные запуски без дублирования строк
- выполняет валидацию полученного набора данных
- формирует простой отчет о готовности данных для дальнейшей ML-работы

# 3. Объем работ этого спринта
Входит в объем

В этот спринт входит следующая работа.

## A. Отдельный модуль подготовки ML-данных

Будет добавлен выделенный модуль для подготовки исторических данных, ориентированных на ML.

Его обязанности:
- загружать конфигурацию
- получать исторические свечи
- сохранять данные в PostgreSQL
- валидировать набор данных
- формировать отчет

## B. Запуск в Docker

Компонент подготовки ML-данных будет запускаться как отдельная Docker-задача, независимо от FastAPI-приложения и панели управления.

Это позволит изолировать путь обучения и подготовки данных от живой торговой системы.

## C. Режим по умолчанию выключен, опциональное выполнение

Компонент не будет запускаться автоматически.

Он будет спроектирован так, чтобы им было удобно управлять вручную в процессе разработки, его можно запускать только при необходимости и пропускать в остальных случаях.

Это особенно полезно при переключении между задачами во время локальной разработки.

## D. Получение исторических данных

Задача будет загружать исторические данные OHLCV-свечей для настраиваемых параметров:
- инструмента
- таймфрейма
- начальной временной метки
- конечной временной метки

Реализация будет повторно использовать существующую логику исторического ingestion там, где это уместно, вместо полной переработки интеграции с биржей.

## E. Выделенная загрузка в PostgreSQL для ML

Загруженные данные свечей будут сохраняться в отдельный экземпляр PostgreSQL, предназначенный для подготовки ML-данных и будущих сценариев обучения.

Эта ML-база будет изолирована от операционного контура основного приложения, чтобы:
- ingestion и подготовка, связанные с ML, не влияли на основную торговую среду
- подготовка исторического набора данных могла развиваться независимо от live-системы
- будущие таблицы признаков, обучающие наборы и артефакты моделей можно было хранить без жесткой связки с торговой базой

Процесс загрузки будет идемпотентным, то есть:
- повторный запуск того же исторического диапазона безопасен
- повторно не вставляются дублирующиеся строки свечей
- повторные запуски можно использовать для обновления или валидации ML-набора данных без порчи существующих данных

## F. Валидация данных

После загрузки компонент выполнит базовые проверки набора данных, такие как:
- ожидаемое и фактическое количество свечей
- покрытие по диапазону временных меток
- пропуски, разрывы в интервалах
- некорректные соотношения OHLC
- некорректные или отрицательные значения объема

## G. Отчет о готовности данных

В конце выполнения задача сформирует простой отчет, в котором будет указано:
- какой диапазон был запрошен
- что было фактически загружено
- были ли найдены разрывы или некорректные строки
- готов ли набор данных к следующему ML-этапу

Этот отчет будет служить операционной контрольной точкой перед началом инженерии признаков.

# 4. Что явно не входит в объем

Чтобы спринт оставался сфокусированным и реалистичным для одной недели, в него не входят следующие пункты:
- инженерия признаков
- генерация меток
- обучение модели XGBoost
- оценка модели
- подбор гиперпараметров
- интеграция инференса в TradingView webhook
- бэктестинг стратегии
- автоматическое переобучение или планирование запусков
- широкое расширение покрытия автоматическими тестами

Эти задачи будут решаться в следующих спринтах после стабилизации конвейера сырых данных.

# 5. Модель выполнения, опциональность

Ключевое требование этого спринта состоит в том, что ML-задача должна быть легко управляемой в процессе разработки.

Планируемое поведение
- ML-задача подготовки данных будет выключена по умолчанию
- она будет запускаться только при явном старте
- ею будет просто управлять во время разработки
- она будет вести себя как ручная одноразовая задача, а не как постоянно работающий сервис

Почему выбран такой подход

Это позволяет избежать лишней фоновой активности и сохранить чистоту live-стека, при этом оставляя удобный способ запускать подготовку ML-данных по мере необходимости.

# 6. Подход к валидации

В этом спринте приоритет будет отдан операционной валидации, а не широкому набору новых автоматических тестов.

Поскольку реализация будет по возможности переиспользовать уже протестированные компоненты ingestion, новая работа сосредоточится на:
- runtime-валидации
- проверках качества набора данных
- опциях dry-run и безопасного запуска, где это полезно
- отчете о готовности после выполнения
- ручной сквозной проверке работы задачи

Это позволит сфокусироваться на создании первого рабочего слоя подготовки ML-данных без чрезмерного расширения объема.

# 8. Результаты поставки

К концу спринта ожидаются следующие артефакты:
- Отдельный модуль подготовки ML-данных
- Docker-упакованная опциональная задача для независимого запуска
- Поддержка конфигурации инструмента, таймфрейма и диапазона дат
- Загрузка исторических данных в PostgreSQL
- Идемпотентное поведение при повторных запусках
- Этап валидации набора данных
- Отчет о готовности данных

# 9. Критерии приемки

Спринт считается завершенным, когда выполнены следующие условия:
- в кодовой базе существует отдельный компонент подготовки ML-данных
- он может запускаться независимо от основного приложения
- он не включен по умолчанию
он может загружать исторические данные свечей для заданного диапазона
он может сохранять эти данные в PostgreSQL
повторный запуск того же диапазона не создает дубликатов
после выполнения он формирует результат валидации и отчет

# 10. Высокоуровневый технический подход

На высоком уровне спринт реализует такой поток:

Ручной запуск, задача подготовки ML-данных, получение исторических рыночных данных, загрузка в PostgreSQL, валидация набора данных, отчет о готовности данных

Этот процесс намеренно отделен от:
- обработки запросов FastAPI
- потокового live-ingestion по WebSocket
- обработки Redis streams
- потока TradingView webhook

# 11. Риски и предположения

Предположения:
- исторические рыночные данные по-прежнему доступны через API биржи
- PostgreSQL доступен, и миграции уже применены
- существующие компоненты ingestion можно переиспользовать как ожидается

Риски:
- ограничение частоты запросов или проблемы доступности внешнего API
- крайние случаи диапазона данных рядом с неполными или открытыми свечами
- потеря времени на настройку контейнера и конфигурации, если появятся проблемы, зависящие от окружения

Эти риски управляемы в рамках объема спринта.
