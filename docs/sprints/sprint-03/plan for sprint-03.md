## 1. Цель спринта

Цель этого спринта - доставить первый самостоятельный компонент конвейера машинного обучения для торговой системы: Dockerized задачу подготовки исторических данных, которую можно запускать независимо от основного приложения.

Этот компонент подготовит необработанный рыночный датасет, который затем будет использоваться для:

- feature engineering
- генерации меток
- обучения модели XGBoost
- экспериментов по проверке сигналов

Цель спринта - создать основу данных.

Перед обучением любой модели машинного обучения системе нужен надежный, воспроизводимый исторический датасет.

Конвейер обучения модели будет зависеть от:

- согласованной истории свечей OHLCV
- стабильных временных диапазонов
- идемпотентных повторных запусков
- проверенного покрытия данных
- процесса, изолированного от живого торгового приложения

Без этого шага последующая работа с ML будет строиться на нестабильных предположениях о необработанных рыночных данных.

Этот спринт создает такую основу.

---

## 2. Планируемый результат

В конце спринта проект будет включать необязательную настройку подготовки ML-данных с двумя отдельными частями:

1. выделенный ML PostgreSQL сервис для хранения подготовленных исторических рыночных данных
2. Dockerized задачу подготовки ML-данных, которая записывает данные в эту базу

Задача подготовки ML-данных будет:

1. по умолчанию отключена
2. запускаться только вручную
3. загружать исторические данные свечей рынка для настраиваемого диапазона
4. сохранять или обновлять данные в выделенной ML PostgreSQL базе
5. поддерживать безопасные повторные запуски без создания дубликатов строк
6. обновлять существующие значения свечей, если биржа предоставляет исправленные данные
7. валидировать итоговый датасет по явным порогам качества
8. формировать отчет о готовности данных, сохраненный как постоянный JSON артефакт
9. завершаться после окончания работы

Выделенная ML PostgreSQL база будет:

1. отдельной от основной базы приложения
2. сохранять данные между запусками подготовки данных
3. использовать собственный Docker volume
4. оставаться доступной для будущих задач feature engineering и обучения
5. останавливаться, когда не нужна, без удаления подготовленных данных

Задача подготовки данных временная.

ML база постоянная.

## 3. Объем этого спринта

### Входит в объем

Этот спринт включает следующую работу:

#### A. Отдельный ML модуль подготовки данных
Будет добавлен отдельный модуль для подготовки исторических данных, ориентированный на ML.

Его ответственность будет заключаться в следующем:

- загрузка конфигурации
- получение исторических свечей
- сохранение данных в выделенную ML PostgreSQL базу
- валидация датасета
- генерация отчета

---

#### B. Выделенный ML PostgreSQL сервис
Будет добавлен отдельный PostgreSQL сервис для подготовки ML-данных и будущих рабочих процессов обучения.

Эта база будет отделена от основной базы приложения.

Ее назначение - хранить ML-данные, не смешивая их с операционными данными торгового приложения.

ML PostgreSQL сервис будет:

- использовать отдельную базу данных и отдельный том хранения
- сохранять данные между запусками
- запускаться только когда нужна подготовка ML-данных или работа по обучению
- оставаться независимым от обычной базы приложения
- не входить в стандартный runtime живой торговли

Остановка контейнера ML PostgreSQL не удалит подготовленные данные.

Данные останутся в volume ML базы, если том не будет явно удален или сброшен.

---

#### C. Выполнение в Docker
Компонент подготовки ML-данных будет работать как отдельная Docker задача, независимая от FastAPI приложения и dashboard.

Это изолирует путь подготовки данных и обучения от живой торговой системы.

Сама задача подготовки ML-данных будет временной.

Она будет:

- запускаться вручную
- загружать и записывать настроенные данные
- валидировать результат
- записывать отчет
- завершаться после окончания работы

Это не означает, что ML база временная.

ML база является stateful компонентом и сохраняет подготовленные данные после завершения задачи.

---

#### D. Модель по умолчанию отключено / опциональный запуск
ML компоненты не будут запускаться по умолчанию.

Они будут спроектированы для удобного ручного управления в разработке, чтобы их можно было запускать только при необходимости и пропускать в остальных случаях.

Это особенно полезно при переключении туда и обратно во время локальной разработки.

Обычный стек приложения не должен автоматически запускать задачу подготовки ML-данных или ML PostgreSQL сервис.

---

#### E. Получение исторических данных
Задача будет загружать исторические OHLCV свечи для настраиваемых:

- инструмента
- таймфрейма
- начального timestamp
- конечного timestamp

Реализация по возможности будет повторно использовать существующий Gate.io клиент исторических свечей и модели сохранения.

Чтобы не дублировать логику, этот спринт может включать небольшое выделение общих helper'ов для сопоставления свечей или сохранения, если это потребуется.

Полный рефакторинг ingestion библиотеки не входит в этот спринт.

Цель - повторно использовать полезные существующие части, не превращая этот спринт в широкомасштабный рефакторинг.

---

#### F. Загрузка в выделенную ML PostgreSQL базу
Загруженные данные свечей будут сохраняться в выделенной ML PostgreSQL базе.

Эта база отдельна от основной базы приложения.

Такое разделение гарантирует, что:

- ML эксперименты не изменяют и не загрязняют основную торговую базу
- подготовка ML данных может развиваться независимо от живой системы
- будущие таблицы признаков, промежуточные датасеты, результаты валидации и данные, связанные с обучением, можно добавлять без привязки к торговой базе
- подготовленные ML данные остаются доступными после завершения задачи подготовки данных
- будущие задачи обучения смогут читать уже подготовленные данные из ML базы

Процесс загрузки будет идемпотентной операцией синхронизации, что означает:

- повторный запуск того же исторического диапазона безопасен
- дубликаты строк свечей не создаются
- если биржа обновила свечу, существующая строка в ML базе может быть обновлена
- загрузка нового диапазона дат добавляет или обновляет только соответствующие свечи
- существующие данные не очищаются, если явно не запрошен reset

Сброс или удаление данных ML базы не является поведением по умолчанию.

Любой разрушительный reset должен быть отдельным явным действием.

---

#### G. Валидация данных с явными порогами
После загрузки компонент провалидирует датасет и присвоит четкий статус:

- **OK** - нет пропущенных интервалов и нет невалидных строк
- **WARNING** - есть небольшие пропуски, но они ниже настраиваемого порога
- **FAILED** - объем пропущенных данных превышает порог, либо найдены невалидные строки OHLC или volume

Проверки валидации будут включать:

- ожидаемое и фактическое количество свечей
- пропущенные интервалы / gaps
- количество пропущенных интервалов
- долю пропущенных интервалов
- фактический начальный timestamp
- фактический конечный timestamp
- невалидные отношения OHLC
- отрицательные или некорректные значения volume
- покрытие запрошенного диапазона дат

Начальные правила статусов:

- `OK` означает, что загруженные данные соответствуют запрошенному диапазону и не найдены невалидные строки
- `WARNING` означает, что есть пропуски, но они ниже настроенного порога
- `FAILED` означает, что объем пропущенных данных превышает настроенный порог, либо найдены невалидные строки OHLC или volume

В качестве начальной границы между WARNING и FAILED будет использоваться настраиваемый порог пропущенных данных, например `0.1%`.

---

#### H. Отчет о готовности данных, сохраненный как артефакт
В конце запуска задача сгенерирует JSON отчет валидации, который будет содержать:

- какой диапазон был запрошен
- что фактически было загружено
- фактическое количество свечей и ожидаемое количество свечей
- количество пропущенных gap
- долю пропущенных gap
- количество невалидных строк
- итоговый статус
- timestamp генерации отчета

Отчет будет сохранен в смонтированную директорию artifacts, например:

```text
artifacts/ml-data-prep/latest.json
artifacts/ml-data-prep/report_BTC_USDT_15m_2024-01-01_2024-06-01.json
```

Это сохраняет отчет доступным после выхода контейнера и позволяет позднее задачам обучения автоматически проверять готовность данных.

Отчет не должен существовать только в логах контейнера.

Логи могут показывать краткую сводку, но JSON отчет должен сохраняться как постоянный артефакт.

## 4. Что явно вне объема

Чтобы спринт оставался сфокусированным и реалистичным на одну неделю, следующие пункты в него не входят:

- feature engineering
- генерация меток
- обучение модели XGBoost
- оценка модели
- настройка гиперпараметров
- интеграция инференса в TradingView webhook
- backtesting стратегии
- автоматическое переобучение или планировщик
- широкое новое покрытие автоматическими тестами
- экспорт в Parquet или DuckDB для feature или training датасетов
- полный редизайн или рефакторинг ingestion библиотеки
- несвязанная техническая уборка долга

Эти задачи будут решаться в следующих спринтах после стабилизации сырого конвейера данных.

Текущий спринт сфокусирован на создании надежной основы из сырых данных.

## 5. Модель выполнения и опциональность

Ключевое требование этого спринта заключается в том, что ML компоненты должны легко управляться в разработке.

### Планируемое поведение
- ML компоненты будут выключены по умолчанию
- обычный стек приложения не будет запускать их автоматически
- ML PostgreSQL сервис можно будет запускать вручную, когда нужна работа с ML
- задачу подготовки ML-данных можно будет запускать только явно
- задача подготовки ML-данных завершится после окончания работы
- подготовленные ML данные останутся в ML PostgreSQL базе после завершения задачи
- ML PostgreSQL базу можно будет останавливать, когда она не нужна
- остановка ML PostgreSQL базы не удалит данные

### Почему выбран такой подход
Это позволяет избежать лишней фоновой активности и сохраняет live trading стек чистым, при этом подготовку ML данных легко запускать при необходимости.

Важно разделение:

- ML база это постоянное хранилище
- задача подготовки ML-данных это временный worker

Это означает, что задачу можно запускать много раз, не теряя ранее подготовленные данные.

Например:

1. запустить ML PostgreSQL сервис
2. запустить задачу подготовки ML-данных для выбранного диапазона дат
3. задача записывает или обновляет свечи в ML базе
4. задача генерирует JSON отчет
5. задача завершается
6. ML база сохраняет данные
7. позже задача обучения может прочитать те же подготовленные данные

Данные ML базы должны удаляться только через явное действие reset.

## 6. Подход к валидации

Этот спринт будет уделять приоритет операционной валидации, а не широкому новому покрытию автоматическими тестами.

Поскольку реализация по возможности будет повторно использовать уже протестированные ingestion компоненты, новая работа сосредоточится на:

- runtime валидации
- проверках качества датасета
- dry-run или безопасных опциях выполнения, где это полезно
- отчете о готовности после завершения
- ручной end-to-end проверке задачи

Это позволяет сфокусироваться на доставке первого рабочего ML слоя подготовки данных без чрезмерного расширения объема.

Отчет валидации будет главным контрольным пунктом качества для этого спринта.

Он сделает результат видимым и измеримым без необходимости в большом новом наборе тестов.

## 8. Критерии приемки

Спринт считается завершенным, когда выполнены следующие условия:

- В кодовой базе существует отдельный ML компонент подготовки данных
- Существует выделенная ML PostgreSQL база отдельно от основной базы приложения
- ML PostgreSQL база сохраняет данные между запусками, используя свой storage volume
- ML компоненты не включены по умолчанию
- Задача подготовки ML-данных может запускаться независимо от основного приложения
- Задача может загружать исторические данные свечей для заданного диапазона
- Задача может сохранять или обновлять эти данные в ML базе
- Повторный запуск того же диапазона не создает дубликатов
- Повторный запуск того же диапазона может обновлять существующие значения свечей при необходимости
- Задача валидирует датасет и формирует отчет с четким статусом
- Отчет сохраняется как постоянный JSON артефакт
- Подготовленные данные остаются доступными после завершения задачи подготовки данных

## 9. Высокоуровневый технический подход

На высоком уровне спринт реализует такой поток:

```text
Ручной запуск -> задача подготовки ML-данных -> получение исторических рыночных данных -> загрузка в ML PostgreSQL -> валидация датасета -> артефакт отчета о готовности данных
```

Задача подготовки ML-данных записывает данные в выделенную ML PostgreSQL базу.

После завершения задача выходит.

ML PostgreSQL база сохраняет подготовленные данные.

Этот процесс намеренно отделен от:

- обработки запросов FastAPI
- live WebSocket ingestion
- обработки Redis stream
- потока TradingView webhook
- основной базы приложения

## 10. Риски и допущения

### Допущения
- Исторические рыночные данные остаются доступными через API биржи
- Выделенный ML PostgreSQL сервис доступен, когда запускается ML задача
- Том данных ML PostgreSQL сохраняется между запусками
- Необходимую схему ML базы можно создать или мигрировать в рамках настройки
- Существующие ingestion компоненты можно переиспользовать ожидаемым образом

### Риски
- Ограничения по запросам к внешнему API или проблемы с доступностью
- Краевые случаи диапазона дат около неполных или открытых свечей
- Потеря времени на настройку контейнеров или конфигурации, если проявятся проблемы, зависящие от окружения
- Небольшое выделение общих helper'ов может занять дополнительное время, если существующий backfill код слишком тесно связан
- Валидация может выявить пропуски в данных биржи, которые нужно будет задокументировать, а не исправлять немедленно

Эти риски управляемы в рамках объема спринта.

## 11. Ожидаемая польза после этого спринта

После этого спринта у проекта появится воспроизводимый и изолированный шаг подготовки исторического датасета для ML работы.

У проекта также появится выделенная ML база, в которой подготовленные исторические рыночные данные смогут сохраняться независимо от основной торговой системы.

Это создаст чистую стартовую точку для следующего спринта, где фокус может перейти на:

- feature engineering
- создание тренировочного датасета
- обучение XGBoost
- оценку модели

Главное преимущество в том, что будущая ML работа не будет начинаться с разовых загрузок данных.

Она будет начинаться с контролируемого процесса подготовки данных с постоянным хранилищем, валидацией и сохраненным отчетом о готовности.
