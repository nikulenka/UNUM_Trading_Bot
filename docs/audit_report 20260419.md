# Отчет по аудиту документации проекта

**Цель аудита:** Проверка полноты, корректности и непротиворечивости технической документации в директории `docs/reference/` на соответствие текущей кодовой базе.

---

## 1. Общая оценка
Документация в `docs/reference/` признана **высококачественной и достоверной**. Это полноценный справочник разработчика, который точно отражает архитектурные решения и детали реализации.

*   **Количество файлов:** 8 (все ключевые аспекты системы охвачены).
*   **Стиль:** Технически точный, с использованием Mermaid-диаграмм и примеров JSON-объектов.
*   **Актуальность:** Документация соответствует состоянию кода на текущий момент (апрель 2026 г.).

---

## 2. Детальные результаты по модулям

### 2.1 Архитектура (`architecture.md`)
*   **Вердикт:** Полное соответствие.
*   **Подтверждено в коде:**
    *   Разделение на процессы (FastAPI, Streamlit, Backfill CLI).
    *   Использование Redis Streams (`market.events`, `system.events`, `signal.events`) как центральной шины данных.
    *   Асинхронная модель потребления данных через `ws_consumer.py`.

### 2.2 API (`api.md`)
*   **Вердикт:** Корректно.
*   **Подтверждено в коде:**
    *   Эндпоинт вебхука `/tradingview/signal` (`app/api/tradingview.py`) использует `TradingViewSignal` Pydantic модель и проверяет `TV_WEBHOOK_SECRET`.
    *   Служебные эндпоинты (`/ready`, `/ingestion/status`) зарегистрированы в `app/main.py`.

### 2.3 Система бэкфилла (`backfill.md`, `backfill-cli.md`)
*   **Вердикт:** Очень подробно и точно.
*   **Подтверждено в коде:**
    *   Логика возобновляемого бэкфилла через таблицу `backfill_states` в PostgreSQL.
    *   Обработка «водяных знаков» (watermarks) и идемпотентная запись свечей (`ON CONFLICT DO NOTHING`).
    *   CLI аргументы (`--instrument`, `--from-date`, `--force`) в `app/modules/ingestion/backfill.py` полностью совпадают с описанием.

### 2.4 Статус фида (`feed-status.md`)
*   **Вердикт:** Соответствует реализации.
*   **Подтверждено в коде:**
    *   Работа `FeedStatusStore` в `app/core/feed_status.py` как потокобезопасного синглтона.
    *   Логика переходов: `live -> stale` при отсутствии данных более 60 секунд.
    *   Особенности метода `check_stale()` (возврат `True`, если данные никогда не поступали).

### 2.5 Интеграция с Gate.io (`gateio-client.md`)
*   **Вердикт:** Корректно.
*   **Подтверждено в коде:**
    *   Поддержка двух форматов ответа (список и словарь) в REST-клиенте.
    *   Использование `Decimal` для финансовых данных.
    *   Логика переподключения (exponential backoff) в WebSocket-клиенте.

## 3. Выявленные особенности и технический долг
Файл `known-limitations.md` честно фиксирует текущие ограничения, что подтверждается кодом:
1.  **Дублирование кода:** Объект `feed_status_store` действительно инициализируется дважды (в `app/core/config.py` и `app/core/feed_status.py`).
2.  **Заглушки:** Эндпоинт `/ready` всегда возвращает `True` через вспомогательную функцию `is_database_connected()`, реальные проверки Postgres/Redis отсутствуют.
3.  **Оптимизация:** Вебхук TradingView создает новое подключение к Redis на каждый запрос, что допустимо для текущей нагрузки, но отмечено как зона роста.
